目录主要文件讲解

  • source：源文件，存放文章

  • themes文件夹：存放主题

  • _config.landscape.yml：主题配置文件

  • _config.yml：hexo配置文件

  • public文件夹：是 Hexo 生成的完整静态网站的根目录，存放博客的“成品”所在的地方。

疑问

为什么我在使用git bash时，初始化hexo执行hexo init就不行，要在前面加上npx，执行npx init hexo就可以，为什么？

核心原因在于 Hexo 的命令行工具 hexo-cli 没有被全局安装，或者它的路径没有被添加到系统的环境变量中

执行以下代码，就可以进行全局安装。安装成功后，系统就能在任何位置识别 hexo 这个命令了。

npx是干什么的？

npx是npm自带的一个工具（从 npm v5.2.0 版本开始），可以把它理解为一个 “临时快递员” 。

  • 作用：npx 会帮你临时安装并执行一个 npm 包里的命令，用完即删，不会污染你的全局环境。

  • 和 npm 的区别：

    • npm 像一个 “超市购物车” ，主要负责把包下载并长久存放在你的电脑里（全局或项目文件夹）。
    • npx 像一个 “外卖员” ，你不必提前囤货，只需要告诉他你需要什么，他会立刻去取来给你用，用完了也不用你收拾。

hexo g

hexo g 是 hexo generate 的简写，它的作用是根据你的 Markdown 文章、主题配置等源文件，生成完整的静态网页到 public 文件夹。

具体功能

  • 读取 source/ 下的所有文章、页面、草稿等 Markdown 文件。
  • 应用主题模板、插件、布局，将其渲染为 HTML 文件。
  • 生成分类、标签、归档等索引页面。
  • 最终输出到 public 目录，这就是最终要部署到服务器的静态网站文件。

hexo s

hexo s 是 hexo server 的简写，它的作用是在本地启动一个开发服务器，让你可以在浏览器中实时预览博客网站。关闭服务器，在 Git Bash 中按 Ctrl + C 即可停止。

主要用途

  • 本地预览：启动后默认访问 http://localhost:4000，你可以看到和线上几乎一样的页面。
  • 热重载：修改文章、主题或配置后，保存文件，浏览器会自动刷新（或手动刷新），方便调试。
  • 无需部署：不用推送到 GitHub 就能看到效果，适合写作和调整样式时使用

启动本地服务器预览。当你运行 hexo s 时，它会自动先执行 hexo g，所以通常不需要单独执行 hexo g。

hexo d

hexo d 是 hexo deploy 的简写，它的作用就是将你的博客“发布”到线上——也就是把你本地生成的静态网站文件推送到远程仓库（比如 GitHub Pages、Gitee Pages 等），让全世界都能访问。

常用方式

  • 单独部署
  • 生成并部署

这个==先执行 hexo g 再执行 hexo d。
